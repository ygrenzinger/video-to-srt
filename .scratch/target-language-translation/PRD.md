# PRD: Target Language Translation Pipeline

Status: ready-for-agent

## Problem Statement

The user can turn a Media Source into timestamped SRT subtitles, but the generated SRT is only in the language returned by the selected Transcription Provider. When the user needs subtitles in another language, they must manually translate the SRT outside the CLI, preserve timestamps themselves, and retry the translation manually if the translation step fails.

The user needs `video-to-srt` to translate Subtitle Cues after transcription, keep timestamps intact, write both the source-language and translated SRT for Media Sources, and allow an existing SRT to be translated again without re-running transcription.

## Solution

Add target-language translation as a transformation step after transcription. When the user passes `--target-language`, the CLI writes the normal source-language SRT and then writes a translated SRT whose filename appends the target language code.

The target language is a simple allowlisted language code, not a regional locale. The accepted codes are:

`ar`, `bn`, `br`, `ca`, `cs`, `da`, `de`, `el`, `en`, `es`, `fa`, `fi`, `fr`, `gu`, `he`, `hi`, `hr`, `id`, `it`, `ja`, `kn`, `ko`, `lo`, `mr`, `ms`, `ne`, `nl`, `no`, `pl`, `pt`, `pa`, `ro`, `ru`, `sr`, `sv`, `ta`, `te`, `th`, `tl`, `tr`, `uk`, `ur`, `vi`, `zh`.

Translation uses a distinct Translation Provider, selected by `--translation-provider mistral|grok`. The translation model is selected by `--translation-model`, with provider-specific defaults. Media Source runs derive a convenient default Translation Provider from the transcription provider where possible. Existing `.srt` files are accepted as Subtitle Sources only when `--target-language` is present, so users can retry translation from a previously created source-language SRT.

## Implementation Decisions

- Add `Target Language` as the user-facing term for the requested translated subtitle language.
- Add `Translation Provider` as a service that translates Subtitle Cue text while preserving cue timing.
- Add `Subtitle Source` as an existing SRT accepted for translation-only retry. Do not expand `Media Source` to include SRT files.
- Add `--target-language` to enable translation. The accepted values are the product-defined simple language-code allowlist from the Solution section.
- Add `--translation-provider` with accepted values `mistral` and `grok`.
- Add `--translation-model` for Translation Provider model overrides.
- Keep `--provider` and `--model` scoped to Transcription Provider behavior.
- Default translation models are `mistral-large-latest` for Mistral and `grok-4.3` for Grok.
- For media runs, default the Translation Provider from the transcription provider: Grok transcription defaults to Grok translation, and Voxtral transcription defaults to Mistral translation.
- For Subtitle Source translation, default the Translation Provider to Mistral.
- Split the transcription flow so Transcription Providers can produce Subtitle Cues before SRT writing. The application orchestration writes the source-language SRT and then optionally writes the translated SRT.
- Translation works on batches of Subtitle Cues, not raw SRT text. The provider receives cue IDs and cue text and must return translated text for the same cue IDs.
- Translation must preserve cue start and end timestamps exactly.
- Translation must fail if the provider response is not valid structured JSON, misses a cue, duplicates a cue, adds an unexpected cue, or returns empty translated text.
- Translation provider calls should use structured JSON output where supported, low randomness, and prompts that explicitly forbid changing cue IDs or timing.
- Media Source translation writes both source-language and translated SRT files.
- Subtitle Source translation writes only the translated SRT file and never overwrites the input SRT.
- On media translation failure after successful transcription, keep the source-language SRT, return non-zero, and do not report success paths in quiet mode.
- Add SRT parsing for Subtitle Source input so existing SRT files can be converted back into Subtitle Cues before translation.
- Add a short ADR because the feature introduces a new pipeline boundary, deliberately separates Translation Provider from Transcription Provider, and introduces Subtitle Source instead of broadening Media Source.
- Update user-facing documentation with flags, accepted language codes, output naming, quiet-mode output, provider defaults, API key requirements, and retry-from-SRT behavior.

## Testing Decisions

- Good tests should verify external behavior and contracts: CLI arguments, output paths, stdout/stderr behavior, generated files, provider request shape, response validation, and failure outcomes.
- Avoid testing private helper details directly when the same behavior can be asserted at an app or adapter boundary.
- App-level orchestration tests are the highest seam for validating target-language argument handling, provider/model defaults, Media Source versus Subtitle Source branching, output naming, quiet output, and partial-output failure behavior.
- Existing app tests with fake runner functions are the prior art for orchestration behavior.
- Transcription provider tests should be adjusted only as needed for the new cue-returning boundary while preserving existing provider behavior.
- Translation provider tests should use local HTTP test servers and fake environment lookups, following the existing provider adapter testing style.
- Translation provider tests should cover default models, custom models, missing API keys, authorization headers, request body shape, structured response parsing, retryable HTTP failures, non-retryable HTTP failures, and network failures.
- Translation validation tests should cover missing cue IDs, duplicate cue IDs, unexpected cue IDs, empty translations, malformed JSON, and successful multi-cue batch translation.
- Subtitle parser tests should cover normal SRT parsing, multi-line cue text, trimming behavior, invalid timestamps, missing indexes, malformed separators, and round trip with SRT formatting.
- CLI validation tests should cover all accepted target language codes and representative rejected values such as empty strings, uppercase variants, region tags, unknown codes, and malformed codes.
- Output tests should verify Media Source translation writes both files and Subtitle Source translation writes only the translated file.
- Quiet-mode tests should verify one stdout line without translation, two stdout lines for Media Source translation, one stdout line for Subtitle Source translation, and silent stderr on successful quiet runs.
- Failure tests should verify a kept source-language SRT when media translation fails and no misleading success output.
- Regression tests should verify existing YouTube Source, Local Video Source, provider, model, cookie, version, and no-translation quiet behavior remain unchanged.
- The final regression command should be `go test ./...`.

## Out of Scope

- Accepting regional locale tags such as `fr-FR`, `en-US`, or `pt-BR`.
- Auto-detecting the source language before translation.
- Guaranteeing provider support or translation quality for every accepted target language code.
- Maintaining a provider-derived language support matrix.
- Translating directly from audio without first producing source-language Subtitle Cues.
- Asking the LLM to rewrite raw SRT files directly.
- Changing Subtitle Cue timing to improve translated readability.
- Splitting or merging Subtitle Cues after translation.
- Adding custom readability limits, line wrapping, or subtitle reflow.
- Batch processing multiple Media Sources or Subtitle Sources.
- Adding providers beyond Mistral and Grok.
- Adding local or offline machine translation.
- Adding a GUI, web app, or interactive translation review mode.
- Creating implementation issues; this PRD is ready for a follow-up issue-splitting pass.

## Further Notes

The language-code allowlist is a product decision, not an exhaustive provider capability claim. Mistral documentation describes Mistral Large as multilingual and especially strong in several European languages; xAI documentation exposes Grok text-generation and structured-output APIs but does not publish an exhaustive text translation language list.

The implementation should be strict about structural correctness and modest about semantic claims. It can guarantee that cue IDs and timestamps are preserved when writing translated SRT, but it cannot locally verify translation quality across all accepted languages.
